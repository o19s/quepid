# Visual parity tooling

Automated **screenshot** and **JSON API shape** comparison between two git branches, using an isolated Docker Compose project so it does not clash with your normal dev stack.

## What it does

1. Creates optional [git worktrees](https://git-scm.com/docs/git-worktree) next to the repo (one per branch).
2. For each branch: `bin/setup_docker`, `yarn build`, then starts **Puma + Solid Queue worker** via a minimal `Procfile.vp` (no `Procfile.dev` asset watchers, which can exit under Foreman and stop the stack).
3. Serves the app through **nginx on [http://localhost:8080](http://localhost:8080)** (Compose override publishes `8080:80`).
4. Runs Playwright against that URL, then diffs API response **key structures** (not values).
5. Writes `test/visual_parity/report.html`.

## Files

| Path | Purpose |
|------|---------|
| `test/visual_parity/run_comparison.sh` | Orchestrator (worktrees, Docker, capture, report) |
| `test/visual_parity/capture_screenshots.mjs` | Playwright full-page screenshots |
| `test/visual_parity/compare_apis.mjs` | Capture / diff API shapes |
| `test/visual_parity/generate_report.mjs` | HTML report |
| `docker-compose.visual-parity.yml` | Override: alternate ports, `Procfile.vp`, nginx config |
| `nginx.vp.conf` | Nginx proxy to the `app` service; sets **`Host: $http_host`** so Rails does not see `app:3000` (blocked by HostAuthorization when using variable `proxy_pass`) |

Generated output is gitignored under `test/visual_parity/screenshots/`, `api_structures/`, and `report.html`.

## Prerequisites

- Docker (Compose v2 with [merge `!override`](https://docs.docker.com/compose/how-tos/multiple-compose-files/merge/) support for port replacement).
- Node 20+ and Yarn on the **host** (scripts run on the host; the app runs in Docker).
- After adding dependencies: `yarn install`, then install browsers. **`run_comparison.sh`** sets `PLAYWRIGHT_BROWSERS_PATH` to `node_modules/.cache/ms-playwright` (under the repo) so Playwright does not use Cursor’s `/tmp/...` sandbox cache, where Chromium is often missing when you run the script in a real terminal. If preflight still fails, run:  
  `cd` to repo root, then `npx playwright install chromium` (same env var is optional; the script auto-installs when needed).

## Usage

From the repo root (the worktree that contains `docker-compose.visual-parity.yml`):

```bash
yarn install
npx playwright install chromium
bash test/visual_parity/run_comparison.sh
```

Default branches are `main` and `deangularjs-incremental`. Override:

```bash
VISUAL_PARITY_BRANCHES="main other-branch" bash test/visual_parity/run_comparison.sh
```

A full run exports **`VISUAL_PARITY_BRANCH_A`** / **`VISUAL_PARITY_BRANCH_B`** so `compare_apis --diff` and `generate_report.mjs` compare **those** two dirs even if older captures remain under `api_structures/` or `screenshots/`. For `--report`, set `VISUAL_PARITY_BRANCHES` the same way if you have more than two dirs.

Explicit CLI (standalone):

```bash
node test/visual_parity/compare_apis.mjs --diff --branch-a main --branch-b feature-x
node test/visual_parity/generate_report.mjs --branch-a main --branch-b feature-x
```

Other modes:

```bash
bash test/visual_parity/run_comparison.sh --capture <branch>   # single branch; uses this repo if checkout matches
bash test/visual_parity/run_comparison.sh --report            # rebuild HTML from existing captures
bash test/visual_parity/run_comparison.sh --remove-worktrees   # remove VP worktrees
```

## Compose / Procfile notes

- Normal dev: `bin/docker` uses `Procfile.dev` (watchers). Visual parity uses **`Procfile.vp`** only inside the VP stack (`web` + `worker`), generated by `run_comparison.sh` so it stays aligned with production-style **`Procfile`** (Puma + jobs).
- `COMPOSE_PROJECT_NAME` defaults to `quepid-vp` during a run so databases and containers do not collide with the default project.
- Readiness checks **`/healthcheck`** (not `/`), since the root path can respond with **403** in some Docker/nginx setups while the app is up. The probe treats **2xx and 3xx** as success (some environments redirect `/healthcheck`). Override with `VISUAL_PARITY_HEALTH_PATH` if needed.
- `bin/setup_docker` brings up Ollama with `docker compose up -d ollama` before `ollama pull`, so it works for both the default project and `quepid-vp`.

## Limitations

Use this harness as a **smoke detector**, not proof of full UX or API parity.

- **Static pages only** — Scripts take full-page screenshots after load. They do not open modals, use dropdowns, submit forms, or hover. Differences in those layers are invisible unless you extend the capture flow (e.g. Playwright `setup` steps).
- **Page list is finite** — Which routes run lives in the capture script’s configuration. Many CRUD screens (new/edit/clone, mapper wizard, some admin/auth/account routes, sparklines, etc.) may be omitted unless someone adds them.
- **API diff is shallow** — Only configured endpoints are compared; structure is inferred from samples (often the first element of a list). **Null vs populated** fields and **empty value encoding** (`""` vs `[]` / `{}`) routinely produce false positives.
- **Single context** — One seeded user and dataset. Admin vs non-admin, team membership, and “already judged” style states can change what renders without indicating a layout regression.
- **Case workspace** — Usually one initial state (e.g. one try, default panels). Alternate tries, expanded results, and open modals are not systematically covered.

To extend coverage, add routes or interaction steps in `test/visual_parity/capture_screenshots.mjs` and endpoints in `test/visual_parity/compare_apis.mjs`, then re-run `run_comparison.sh`.
